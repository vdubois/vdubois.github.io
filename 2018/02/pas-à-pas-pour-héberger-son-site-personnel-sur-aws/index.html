Pas à pas pour héberger son site personnel sur AWS

@Vincent Dubois · Feb 12, 2018 · 8 min. de lecture

Dans cet article, j’aimerais partager la méthode que j’ai employé pour mettre en ligne mon site personnel sur AWS. En effet, j’ai eu du mal à trouver des articles complets sur la toile afin de paramétrer tout ça. Finalement, avec un peu de patience et de persévérance, je suis arrivé à mes fins.

Nom de domaine

La première étape a été l’achat du nom de domaine. J’ai choisi OVH car c’est un des registrars les moins chers, et aussi car il m’a été conseillé par un collègue freelance qui a l’habitude d’utiliser OVH pour ses clients. L’interface d’administration notamment est pas mal, elle est assez simple à utiliser.

Préparation du contenu côté AWS : créer un bucket dans S3

Dans tous les articles que j’ai pu trouver, il est conseillé de créer un bucket dans S3 dont le nom correspond exactement au nom de domaine du site que l’on souhaite héberger. Je pense que ce n’est pas une obligation, car on ne passe pas directement par S3, mais par Cloudfront, qui se positionne devant le bucket pour diriger le traffic.

Néanmoins, cela permet de “réserver” le nom de bucket dans S3 car les noms sont uniques à l’international. Cela permet donc par la suite de ne pas se faire voler le nom de bucket qui, quelquepart, nous revient.

Pour réaliser cette opération :

  • Se connecter à la console AWS
  • Aller dans S3
  • Cliquer sur “Créer un compartiment”
  • Choisir un nom de compartiment (voir paragraphe ci-dessus) ainsi qu’une région géographique et cliquer sur “Créer”

Préparation du contenu côté AWS : uploader le contenu et le rendre public

Pour automatiser l’upload des ressources statiques dans S3, j’utilise serverless avec le plugin serverless-finch, ce plugin permet la mise à jour des sources, mais aussi l’exposition du bucket en tant que site web avec les bons droits.

Installer serverless en global et le plugin en local :

npm install -g serverless
npm install --save serverless-finch

Voici le fichier serverless.yml minimal permettant de réaliser l’opération :

service: monsite.fr

provider:
  name: aws
  region: eu-west-1

plugins:
  - serverless-finch

custom:
  client:
    bucketName: monsite.fr
    distributionFolder: dist

Pour déployer le site, taper la commande :

serverless client deploy

Le site est déployé, on peut le vérifier en accédant à l’URL (http://monsite.fr.s3-website-eu-west-1.amazonaws.com/). On peut passer à l’étape suivante.

Créer la distribution Cloudfront

Amazon S3 permet d’héberger des sites statiques, mais dès qu’il s’agit de gérer la moindre règle de redirection ou le moins code d’erreur, ce n’est vraiment pas prévu pour. Amazon, pour rendre ce service (et bien plus encore) propose Cloudfront. Bonne nouvelle pour nous, encore un service gratuit.

Il faut donc se rendre dans la console Amazon dans la partie Cloudfront.

Cliquer sur “Créer une distribution”, puis “Mise en route” dans la partie “Web”.

Remplir le formulaire de création de la distribution :

  • Dans le champ “Chemin d’origine”, sélectionner le bucket S3 créé au préalable (il est présenté sous forme d’URL).
  • Dans le champ “Mise en cache d’un objet”, sélectionner “Personnaliser” et positionner les trois champs suivants à 0. Cela permet de ne jamais garder en cache des ressources du site.
  • Dans “Paramètres de distribution” > “Autres noms de domaine (CNAME)”, saisir le nom de domaine sans le “http(s)://” devant.
  • Laisser le certificat cloudfront par défaut pour l’instant.
  • Plus bas, dans “Objet racine par défaut”, saisir “index.html” afin que le fichier index.html soit utilisé quand vous accèderez à la racine de votre site.
  • Cliquer sur “Créer une distribution”

Le temps de création est très long (compter 10 à 20 minutes).

Il nous manque un petit détail. Il est nécessaire de créer une redirection vers la racine du site lors d’une erreur 404 car celle que fournit S3 par défaut n’est pas des plus réussies.

Cliquer sur la distribution nouvellement créée dans la liste des distributions Cloudfront, aller sur l’onglet “Pages d’erreurs”. Cliquer sur le bouton “Créer une réponse d’erreur personnalisée”. Dans le formulaire qui s’affiche, choisir “404” dans “Code d’erreur HTTP”, sélectionner “Oui” dans “Personnaliser une réponse d’erreur”. Enfin, saisir “/index.html” dans “Chemin de la page de réponse” et sélectionner “200” dans “Code de réponse HTTP”. Valider la création.

De nouveau, attendre 10 à 20 minutes que Cloudfront prenne en compte la modification.

Faire pointer le nom de domaine vers AWS

Amazon propose un service pour router un nom de domaine vers des sites hébergés dans leur infrastructure. Ce service s’appelle Route 53. En fait, ce service fait beaucoup plus de choses, c’est notamment un registrar reconnu, depuis qu’il s’est associé avec Ghandi.

Une nouvelle fois, il faut se rendre dans la console Amazon, dans la partie Route 53 désormais.

Dans le menu principal, cliquer sur “Zones hébergées”, puis sur le bouton “Créer une zone hébergée”. Dans la partie droite de l’écran, un formulaire s’ouvre. Remplir avec le nom de domaine souhaité (ex. : monsite.fr). Cliquer sur “Créer”.

Attention : Amazon facture 0.50 dollar la création d’une zone hébergée.

Amazon vient de créer une zone hébergée avec le nom de domaine, mais il vient surtout d’y associer 4 serveurs DNS qui vont servir à rediriger le nom de domaine vers AWS. En attendant de configurer cette partie dans OVH, il faut dire à Route 53 vers quoi router le traffic du nom de domaine. Pour cela, cliquer sur “Créer un jeu d’enregistrements”. Si vous souhaitez un préfixe (genre www.monsite.fr) alors il faut saisir les ‘www’ à ce moment là. Sélectionner le type “A - Adresse IPv4”. Dans “Alias”, sélectionner “Oui” et choisir la distribution Cloudfront créée au préalable. Cliquer sur “Créer”. Renouveler l’opération de création d’un jeu d’enregistrements en sélectionnant “AAAA - Adresse IPv6”.

Tout est prêt côté Amazon pour recevoir le traffic. La fin de la configuration se passe dorénavant dans votre registrar.

Configuration des DNS Amazon dans la console OVH

Si comme moi vous êtes chez OVH, rendez-vous dans la console d’administration afin de paramétrer les DNS Amazon.

Dans le menu, cliquer sur Domaine > monsite.fr. Aller dans l’onglet “Serveurs DNS”. Cliquer sur le bouton “Modifier les serveurs DNS”. Dans le formulaire qui apparaît, saisir les 4 serveurs DNS fournis dans la configuration de Route 53. Cliquer sur “Appliquer la configuration”.

Soyez très patients. Cette étape peut durer jusqu’à 48h. A cette issue, votre site est disponible !!

Disclaimer : j’ai pris le parti ici de supprimer les DNS OVH et de ne renseigner que ceux d’Amazon. Cela n’engage que moi de procéder ainsi, car je n’ai qu’un nom de domaine, et qu’il pointe sur Amazon.

Et maintenant, finissons en beauté en rajoutant le HTTPS

Générer le certificat

Votre nouveau site tout nouveau tout beau est disponible mais il n’a pas le HTTPS ?

Plus précisément, il a le HTTPS, mais avec un logo rouge, couleur de l’erreur, ou un https bien barré en rouge qui ne prête pas à confusion. Pas sympa.

Ce serait dommage de s’arrêter là, alors qu’il n’y a plus besoin désormais de payer des fortunes pour avoir son certificat HTTPS rattaché à son nom de domaine. Si vous imaginiez vous générer vous même votre certificat, passez votre chemin, ce n’est pas une bonne idée.

Depuis quelques années, il est tout à fait possible d’avoir des certificats HTTPS reconnus dans les principaux navigateurs grâce à Let’s encrypt. La seule contrainte pour pouvoir générer un certificat est de prouver à Let’s encrypt que vous êtes le propriétaire de votre nom de domaine. Pour cette raison, les étapes précédentes de ce pas à pas doivent être complètement terminées.

Bien évidemment je suis loin d’être le premier à demander un certificat Let’s encrypt pour Cloudfront, il existe des outils qui permettent d’automatiser la création. J’ai personnellement suivi cet article pour créer mon certificat.

Voici les étapes à réaliser :

  • Tout d’abord, il faut installer l’outillage Python nécessaire au script :
pip install letsencrypt && pip install letsencrypt-s3front
  • Avant de passer à l’étape suivante, si comme moi vous n’êtes root, passer en mode root avec la commande :
sudo su
  • Créer le certificat avec la commande suivante :
AWS_ACCESS_KEY_ID=<ACCESS_KEY> \
AWS_SECRET_ACCESS_KEY=<SECRET_ACCESS_KEY> \
letsencrypt --agree-tos -a letsencrypt-s3front:auth \
-i letsencrypt-s3front:installer \
--letsencrypt-s3front:auth-s3-bucket <BUCKET> \
--letsencrypt-s3front:auth-s3-region <REGION> \
--letsencrypt-s3front:installer-cf-distribution-id \<DISTRIBUTION_ID> -d <DOMAIN>

Cette opération négocie avec Let’s encrypt pour générer le certificat en prouvant à l’aide du dépôt d’un fichier sur votre bucket que le domaine est bien le vôtre. Cela n’installe pas le certificat sur AWS cependant.

La commande tapée ci-dessus, une fois finie, donne les chemins d’accès aux différentes parties du certificat (fichiers fullchain.pem et privkey.pem). Mettez-les de côté, on va en avoir besoin juste après.

Importer le certificat

Se connecter à la console Amazon. Aller dans “Certificate Manager”. Cliquer sur “Démarrage”. Une nouvelle page s’ouvre dans laquelle il faut cliquer sur “Importer un certificat”.

Un formulaire de saisie apparaît :

  • Dans “Corps du certificat”, saisir le contenu de la première partie du fichier fullchain.pem généré dans l’étape précédente.
  • Dans “Clé privée du certificat”, saisir le contenu du fichier privkey.pem généré dans l’étape précédente.
  • Dans “Chaine de certificat”, saisir le contenu de la deuxième partie du fichier fullchain.pem généré dans l’étape précédente.

Cliquer sur vérifier et importer. Valider.

Retourner dans Cloudfront. Sélectionner la distribution correspondant au site monsite.fr. Cliquer sur modifier. Dans la partie “Certificat SSL”, le certificat nouvellement importé doit apparaître. Le sélectionner. Valider en cliquant sur “Oui, Modifier”.

La distribution Cloudfront se redéploie en appliquant le certificat Let’s encrypt. Prévoir quelques mot croisés, le temps d’attente est vraiment très long cette fois-ci. Quand Cloudfront affiche que la distribution est déployée, lancez votre URL en HTTPS, vous avez un beau certificat HTTPS valide !

Conclusion

Il manque une pierre à l’édifice que je n’aborderai pas dans cet article, c’est l’automatisation de la génération du certificat. Les certificats Let’s encrypt sont certes gratuits, mais ils ne sont valides que pendant 3 mois. Il ne faut donc pas oublier de les renouveler.

En ce qui concerne AWS, on peut se permettre d’héberger son site personnel, c’est relativement facile à mettre en oeuvre, pour une somme très faible (0.5 dollar par nom de domaine). Les requêtes DNS sont payantes également, compter quelques centimes d’euros par mois. Attention toutefois, si votre site commence à avoir du traffic, alors cela pourrait vous coûter plus cher. En effet, Cloudfront est gratuit jusqu’à 2 millions de requêtes sur vos ressources statiques.
