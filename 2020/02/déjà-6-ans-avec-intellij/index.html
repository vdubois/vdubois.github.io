Déjà 6 ans avec IntelliJ

@Vincent Dubois · Feb 29, 2020 · 7 min. de lecture

Cela fait six ans que j’utilise IntelliJ Idea, aussi bien dans le cadre privé que dans le cadre professionnel.

Après dix années passés à utiliser Eclipse, cela a été un changement conséquent, mais je ne regrette absolument pas. Ce que je regrette en revanche, c’est de ne pas avoir découvert IntelliJ plus tôt.

IntelliJ, c’est avant tout un IDE construit autour d’une cohérence globale. Quand on utilise une fonctionnalité dans un contexte (quand on code en Java par exemple), on pourra retrouver un équivalent dans un autre contexte. C’est pour moi ce qui en fait un outil très intelligent.

Les choses qui m’ont décidé à utiliser IntelliJ

Tous les “grands” l’utilisent

Quand j’ai commencé à m’intéresser à IntelliJ par le biais de mes collègues de bureau, j’ai aussi fait l’expérience de ma première participation à Devoxx France. Et je me suis aperçu que 95% des speakers l’utilisaient.

De plus, une conférence “42 IntelliJ IDEA tips and tricks in 45 minutes” par Hadi Hariri, évangélisateur chez Jetbrains, a fini de me convaincre.

Ça marche

Ça paraît fou de devoir dire ça, mais IntelliJ, ça marche. Cela veut dire pas de bugs, et donc on peut vraiment se focaliser sur l’essentiel : apporter de la valeur dans nos projets.

Une productivité accrue

Quand on commence à utiliser IntelliJ, on découvre un nouveau monde où connaître tous les raccourcis clavier devient un espèce de Graal. On devient alors méga productif par rapport à avant. Les complétions intelligentes, refactorings, et autres fonctionnalités proposées par IntelliJ au développeur vont accroître sa productivité de façon très significative.

Le fait que l’on puisse tout faire dans le même IDE (backend, frontend, gestion de bases de données, …) est également un gain de temps non négligeable. On ne passe plus son temps à changer d’outil.

L’IDE est livré avec un guide de productivité qui permet de savoir votre niveau de productivité ainsi que les statistiques afférentes. C’est aussi un excellent moyen de découvrir des fonctionnalités que vous n’utilisez pas ou que vous n’avez pas assez pris l’habitude d’utiliser.

L’intégration avancée de frameworks

Que vous soyez développeur frontend, backend ou les deux, IntelliJ a toujours un train d’avance sur l’intégration des langages et frameworks du marché. Il est toujours à jour par exemple par rapport aux dernières versions de Java, Spring Boot, mais propose aussi une intégration excellente de frameworks plus récents, comme Quarkus. L’intégration des frameworks tels qu’Angular, VueJS par exemple est très bien faite et permet d'être plus productif.

Les fonctionnalités que j’utilise le plus

Auto-complétions

L’auto complétion intelligente, la complétion de l’instruction en cours et la complétion postfix sont les complétions que j’utilise le plus au quotidien.

Extractions

Extraire une variable, une constante, une méthode, un paramètre de méthode ou un attribut de classe est une fonctionnalité que j’utilise également beaucoup au quotidien. Gain de temps garanti.

Intégration GIT

L’intégration GIT est très avancée. Par exemple, si vous avez un push rejeté car quelqu’un d’autre a poussé des modifications distantes avant vous, IntelliJ le détecte et vous propose de le traiter en merge ou en rebase.

Les “killer features” qui à mon sens en font le meilleur IDE

Le debugger

Le debugger permet :

  • de poser des points d’arrêt dans des lambdas (opérations intermédiaires de streams par exemple)
  • de voir en temps réel le contenu des différentes variables dans l'éditeur
  • depuis une version récente, de deviner ce qui va se passer après le point d’arrêt en cours (Data Flow Analysis)

Décompilateur

On n’a pas toujours accès aux sources des dépendances que l’on utilise au sein de nos projets. Mais pas de problème, car IntelliJ est livré avec un décompilateur qui va vous décompiler les classes à la volée.

Cette fonctionnalité est très puissante, car vous pouvez alors poser des points d’arrêt dans le code décompilé, et le debugger s’arrêtera dans le code qui sera synchronisé à l’exécution. Comme si vous aviez compilé le code au sein de l’application.

La complétion et la documentation partout

Quel que soit votre contexte (édition d’un pom.xml Maven, d’une classe Java, d’un fichier HTML, …) vous avez accès à un tas de complétions ou de documentations pertinentes dans le contexte.

Envie de savoir quels paramètres vous pouvez mettre dans la configuration d’un plugin Maven ? Ajoutez des nouveaux tags sous le tag de configuration et vous le saurez. Envie de savoir à quoi correspondent ces paramètres ? Lancez le raccourci de documentation quand vous êtes dessus et celle ci s’affiche dans une popup.

On ne s’attend pas forcément à avoir de la complétion dans les dialogues de configuration des outils ou dans la boite de dialogue de renseignement du commit dans l’intégration GIT. Et bien pourtant, cela existe !

Plus besoin d’attendre l’exécution de votre code pour voir vos erreurs

IntelliJ vous met en évidence vos erreurs. Que ce soit une dépendance Maven qui n’existe pas, une dépendance Spring qui ne pourra pas être injectée, un mapping Hibernate qui ne pourra pas se faire, vous allez le savoir directement dans l'éditeur.

Cette façon de relever les erreurs est un gain de temps inestimable.

Le double shift

En déclenchant le raccourci clavier shift+shift, vous avez accès à toutes les fonctionnalités de l’IDE, c’est à dire ouvrir un fichier, comme changer le thème, ou encore lancer vos tests unitaires.

J’adore utiliser cette fonctionnalité par exemple pour ouvrir le contenu d’une table dans une base de données : je fais double shift, je tape le nom de la table et je valide avec entrée pour ouvrir.

Language injection

IntelliJ propose une fonctionnalité qui permet d’injecter un type de langage différent du contexte dans lequel on se trouve.

Par exemple, il est possible d’injecter du SQL dans une chaine de caractères Java, et d’avoir alors accès à la complétion des tables, colonnes et mots-clés de la base de données. Cette fonctionnalité peut être déclenchée à n’importe quel moment en faisant Alt+Entrée > Inject language.

L’IDE est assez intelligent pour détecter dans quel contexte injecter automatiquement un autre langage. Par exemple :

  • injecter automatiquement du langage SQL lorsqu’on utilise un PreparedStatement en Java
  • injecter automatiquement du langage Regexp quand on fait un new Regexp en Javascript

Vous voulez copier-coller du JSON dans une chaine de caractères Java ? Pas de panique, IntelliJ va ajouter tous les antislashs nécessaires à la volée. Besoin de modifier le JSON en question ? Faites Alt+Entrée > Inject language > JSON et modifiez directement le JSON dans une fenêtre à part. La chaine de caractères est impactée en temps réel. Pratique !

Des fonctionnalités cachées

Testeur de Regexp

Si vous êtes comme moi et que les Regexp, ce n’est pas votre tasse de thé, sachez que ce n’est pas une fatalité ! Vous pouvez tester vos Regexp directement depuis IntelliJ, grâce à la fonctionnalité d’injection de langage.

Comparaison de jars ou zips

Découverte par hasard, cette fonctionnalité m’a déjà permis de comparer deux versions de jars d’une même librairie. Cette fonctionnalité permet d’aller jusqu'à voir le différentiel de classes décompilées dans le jar !

Oui, mais c’est payant, non ?

Oui, et c’est ce qui rebute les développeurs en général. Ils ne comprennent pas dans un premier temps que l’IDE va leur rapporter plus à long terme que le prix de la license. Cela va les rendre plus productifs, et donc plus ‘bankable’.

Personnellement, je trouve que le temps gagné en un mois, voire moins, pour un développeur permet de rembourser largement le prix de la license.

Conclusion

Lancez vous si ce n’est pas fait ! Vous ne le regretterez pas. Personnellement, je trouve que c’est un grand confort de ne jamais avoir à pester contre son IDE parce qu’il buggue ou qu’il vous freine dans vos taches quotidiennes. Je ne reviendrai pour rien en arrière, je ne peux plus m’en passer !
